# Roamie Pro — Project Report

**An AI-Powered Travel Companion built with the AI-DLC (AI Development Life Cycle) workflow**

---

## 1. Executive Summary

Roamie Pro (codename: Travellerftye) is a full-stack, end-to-end travel management platform. It lets a traveler plan a trip end-to-end — from a 7-question onboarding flow to an AI-generated, budget-aware, weather-aware itinerary, complete with real flight/hotel booking suggestions, receipt OCR expense tracking, flight-disruption rerouting with SMS/WhatsApp alerts, visa requirement checks, packing lists, and live translation.

Every intelligent feature is powered by **AWS Bedrock (Claude Sonnet 4.6)** with a graceful fallback chain to **Claude Haiku 4.5** and then a local **Ollama** model. All data shown to the user is real — generated by the LLM or fetched from free public APIs (Open-Meteo for weather, Nominatim/OpenStreetMap for geocoding). There is no fabricated/random data in the primary path.

| Attribute | Value |
|-----------|-------|
| Project type | Brownfield (extended an existing codebase) |
| Architecture | Monorepo (Express API + React/Vite SPA) |
| Primary language | TypeScript |
| AI runtime | AWS Bedrock — Claude Sonnet 4.6 → Haiku 4.5 → Ollama (fallback chain) |
| Database | SQLite (dev) via Prisma ORM; PostgreSQL-ready for production |
| Cost to run | $0 (Bedrock via hackathon, all other APIs free-tier) |
| Repository | https://github.com/Sansyuh06/roamie.ai-ideaverse_hackathon |

## 2. The AI-DLC Workflow

This project was built and documented using the **AWS AI Development Life Cycle (AI-DLC)** — a structured, stage-gated workflow where an AI agent intelligently selects which phases add value, produces artifacts for each, and logs every interaction in an audit trail.

### 2.2 How AI-DLC Shaped the Build

1. **Workspace Detection** — The agent scanned the repo, identified it as a brownfield TypeScript monorepo (Express + React), and chose the brownfield path requiring reverse engineering.

2. **Reverse Engineering** — Before writing any new code, the agent produced 8 documentation artifacts under `aidlc-docs/inception/reverse-engineering/`:
   - `business-overview.md` — business transactions & domain dictionary
   - `architecture.md` — Clean/Hexagonal architecture diagram & data flows
   - `code-structure.md` — module-by-module breakdown
   - `api-documentation.md` — endpoint inventory
   - `component-inventory.md`
   - `technology-stack.md`
   - `dependencies.md`
   - `code-quality-assessment.md` — identified the syntax error in `DisruptionCoordinator.ts`, demo credentials, and tech-debt items that were later fixed

3. **Requirements Analysis** — Captured the detailed feature specs (LLM adapter, landing page, onboarding chatbot, itinerary generation, disruption shield, receipt OCR, packing, safety, translator, memory substrate).

4. **Construction / Code Generation** — Implemented each feature, verifying TypeScript compilation after every change and logging the work in `aidlc-docs/audit.md`.

```
┌──────────────────────────────────────────────────────────┐
│                  Frontend — apps/web                       │
│   React 19 · Vite 8 · Tailwind 4 · Framer Motion · Zustand │
│   Leaflet maps · Recharts · i18next · React Router 7       │
└───────────────────────────┬──────────────────────────────┘
                            │  HTTP / JSON  (Vite proxy → :3001)
                            ▼
┌──────────────────────────────────────────────────────────┐
│                   Backend — apps/api                       │
│                  Express + TypeScript                      │
├──────────────────────────────────────────────────────────┤
│ Infrastructure: server, config, JWT auth, rate-limit, i18n │
│ Adapters:       controllers · repositories · services      │
│ Use-Cases:      business orchestration                     │
│ Domain:         entities + interface contracts             │
│ Agents:         OpenClaw flight agent, disruption coord.   │
│ Substrate:      3-tier memory (Atoms → Patterns)           │
└──────────┬───────────────────────────────┬────────────────┘
           │                               │
           ▼                               ▼
┌────────────────────┐         ┌───────────────────────────┐
│  AWS Bedrock        │         │  Free Public APIs          │
│  • Claude Sonnet 4.6│         │  • Open-Meteo (weather)    │
│  • Claude Haiku 4.5 │         │  • Nominatim (geocoding)   │
│  • Titan Embeddings │         │  • CallMeBot/Fast2SMS (SMS)│
│  AWS S3 (receipts)  │         │  • Booking/Skyscanner links│
└────────────────────┘         └───────────────────────────┘
           │
           ▼
┌────────────────────┐
│  Ollama (local)     │  ← final fallback if Bedrock unavailable
└────────────────────┘
           │
           ▼
┌────────────────────┐
│  SQLite via Prisma  │  (PostgreSQL-ready)
└────────────────────┘
```

The backend follows **Clean / Hexagonal Architecture**: the domain layer has zero dependencies; use-cases depend only on domain interfaces; adapters implement those interfaces. This is why swapping the AI provider (Anthropic direct → Bedrock) required changing only one adapter.

## 7. The LLM Fallback Chain

The single most important architectural decision. Every AI call routes through `LLMAdapter`:

```
1. AWS Bedrock — Claude Sonnet 4.6   (us.anthropic.claude-sonnet-4-6)
        ↓ on failure / quota
2. AWS Bedrock — Claude Haiku 4.5    (us.anthropic.claude-haiku-4-5-20251001-v1:0)
        ↓ on failure
3. Ollama (local mistral)            — offline fallback
        ↓ on failure
4. Static templates / Nominatim POIs — last resort
```

This guarantees the app never hard-fails on an AI outage, and credentials are resolved by `awsCredentials.ts` (env-var chain in CI/Docker, named profile in local dev — no `ProfileNotFound` crashes).

## 8. Engineering Quality & Fixes

Key issues identified (some via the AI-DLC reverse-engineering quality assessment) and resolved:

- Fixed critical auth bug (`req.user.id` → `req.userId`) that was returning 401 on every v2/memory request.
- Consolidated all Prisma usage onto a single shared singleton (no connection-pool duplication).
- Lazy S3 client init + shared AWS credential resolver (boots cleanly in CI/Docker).
- Removed hardcoded secrets (S3 bucket, phone number) — now env-only, `.env` gitignored.
- Added `MemoryAtom` / `MemoryPattern` models + migration so the memory system persists.
- Fixed JSON-truncation on itinerary generation (raised token limit, concise prompt).
- Forced English output (was localizing to destination language).
- Removed the `objectkki` syntax error in `DisruptionCoordinator.ts`.

Every change was TypeScript compile-verified (`tsc --noEmit`) before commit.

## 9. How to Run

```bash
# Local development
npm install
cp .env.example apps/api/.env        # fill in AWS + secrets
cd apps/api && npx prisma migrate dev && cd ../..
npm run dev                          # API :3001 + Web :5173

# Open: http://localhost:5173/Roamie-Assistant/

# Docker (Postgres + Ollama + API + Web)
docker compose up -d
```

**AWS credentials** (for Bedrock) live in `~/.aws/credentials` under the profile named by `AWS_PROFILE`, or as `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` / `AWS_SESSION_TOKEN` env vars.
